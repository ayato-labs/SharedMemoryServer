# SharedMemoryServer 設計思想 (Design Philosophy)

`SharedMemoryServer` は、単なるデータストレージではありません。複数の異なるAI（独自エージェント、Cursor、Antigravity、Roo Code等）が、あたかも一つの「共有された脳」を持つかのように協調するための**共通基盤（コネクティブ・ティッシュ）**です。

---

## 1. 実現すること (What it Achieves)

### **A. ツールを跨ぐ「文脈の永続性と同期」**
- **永続性**: セッションが終了しても、エージェントが再起動しても、過去の思考プロセスやプロジェクトの進捗を失わない。
- **同期**: Cursorでの修正内容を独自エージェントが即座に理解し、独自エージェントの検証結果をCursorが活用できる「共通の記憶」を提供する。

### **B. 知識の二重構造 (Hybrid Structure)**
- **Knowledge Graph (構造的な知)**: エンティティ、関係、観察に基づいた「硬い知識」。システムのアーキテクチャや依存関係を厳格に定義・検索可能にする。
- **Memory Bank (流動的な知)**: Markdown形式の「柔らかい文脈」。人間とAIが共に読み書きし、プロジェクトの「今」の空気感や設計意図を共有する。

### **C. AIエージェントの「軽量化」と「専門化」**
- エージェント自身が巨大なコンテキストを抱え込む必要をなくし、必要な時に必要な記憶をMCP経由で取り出す「オンデマンド・メモリ」を実現する。

---

## 2. 実現しないこと (Non-Objectives / What it is NOT)

### **❌ 自律的な推論 (Reasoning)**
- このサーバーは「思考」しません。あくまで「記録」と「提供」に徹します。推論や意思決定は、それを利用するエージェント側の役割です。

### **❌ コードの実装保持や実行 (Implementation/Execution)**
- コードの具体的な「実装内容（ソースコード）」を資産として管理するのは LogicHive です。このサーバーは、それらが「プロジェクトのどこで、何の目的で使われており、どのような課題があるか」という**意味論（セマンティクス）とプロジェクト文脈**を専門的に記憶します。

### **❌ 汎用的なデータベース (General-Purpose Database)**
- 何でも保存する DB ではありません。「ソフトウェア開発」と「プロジェクト理解」に特化した、コンテキスト管理のための専用インターフェースです。

### **❌ バージョン管理の代替 (Version Control Replacement)**
- Git の代わりにはなりません。Git は「成果物の履歴」を管理し、SharedMemoryServer は「開発者の思考と文脈の履歴」を補完します。

### **類似知識の自動統合**
マージ・エンジン（動的統合）: 定期的にエンティティ間の類似度を計り、重複した知識をAIが自動で統合・整理する「バックグラウンド整備」機能は作成しません。その理由はハイリスクローリターンだからです。このようにマージするくらいなら検索システムをグレードアップしたほうが効率が良いと考えます。

---

## 3. 基本原則 (Core Principles)

1. **Simple is Best (シンプル・イズ・ベスト)**
   - インターフェースは極力単純に保ち、どんなエージェントからも呼び出しやすくする。
2. **Garbage In, Garbage Out (GIGO) の防止**
   - Knowledge Graph には型を持たせ、無秩序な情報の蓄積を防ぐ。
3. **Connective Tissue (結合組織)**
   - AIとAI、そして人間を結びつけるための「ハブ」として機能する。
